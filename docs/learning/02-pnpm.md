# pnpm

## What is it?

pnpm is a **package manager** - a tool that downloads and manages code libraries (called "packages") that other people have written so you don't have to reinvent the wheel.

When you need to send emails, talk to a database, or validate input, you don't write that code from scratch. You install a *package* that already does it. pnpm is the tool that handles that.

There are three package managers for JavaScript: `npm`, `yarn`, and `pnpm`. They all do the same job. We use **pnpm** because:
- It's **faster** (doesn't re-download packages you already have)
- It uses **less disk space** (shares packages between projects)
- It's **stricter** (prevents subtle bugs that npm/yarn allow)

## Core Concepts

### 1. package.json

Every project has a `package.json` file. It's the project's "identity card" - it lists the project's name, version, and most importantly, its **dependencies** (the packages it needs).

```json
{
  "name": "freediving-app-backend",
  "version": "1.0.0",
  "dependencies": {
    "@nestjs/core": "^10.0.0",
    "prisma": "^5.0.0"
  },
  "devDependencies": {
    "typescript": "^5.0.0",
    "jest": "^29.0.0"
  },
  "scripts": {
    "start:dev": "nest start --watch",
    "build": "nest build",
    "test": "jest"
  }
}
```

Key sections:
- **dependencies**: Packages needed to *run* the app (deployed to production)
- **devDependencies**: Packages needed only during *development* (testing, linting, etc.)
- **scripts**: Shortcuts for commands you run often

### 2. pnpm-lock.yaml

When you install packages, pnpm creates a `pnpm-lock.yaml` file. This records the *exact* version of every package installed. It ensures everyone on the team (and the server) gets identical packages.

**Never edit this file manually.** It's auto-generated. **Always commit it to git.**

### 3. node_modules/

This is the folder where downloaded packages actually live. It's auto-generated by `pnpm install`.

**Never edit files in here.** **Never commit it to git** (it's in `.gitignore`).

### 4. Workspaces (Monorepo)

Our project is a **monorepo** - multiple apps in one repo (backend + mobile). pnpm workspaces let each app have its own dependencies while sharing common ones.

Configured in `pnpm-workspace.yaml`:
```yaml
packages:
  - 'apps/*'
```

This tells pnpm: "there are sub-projects inside the `apps/` folder."

## Essential Commands

```bash
# Install all dependencies (run this after cloning or when someone updates package.json)
pnpm install

# Add a new package to the current project
pnpm add express              # adds to dependencies
pnpm add -D jest              # adds to devDependencies (-D = dev only)

# Remove a package
pnpm remove express

# Run a script defined in package.json
pnpm start:dev                # runs "start:dev" script
pnpm test                     # runs "test" script
pnpm build                    # runs "build" script

# Run a command in a specific workspace (sub-project)
pnpm --filter backend add prisma       # add prisma to just the backend
pnpm --filter mobile start             # start just the mobile app

# Update packages
pnpm update                   # update all packages to latest compatible versions

# See what's installed
pnpm list                     # show installed packages
pnpm list --depth=0           # show only top-level packages (less noise)
```

## The ^ Symbol in Versions

In `package.json` you'll see versions like `"^10.0.0"`. The `^` means "compatible with":
- `^10.0.0` = any version from 10.0.0 up to (but not including) 11.0.0
- `^5.2.1` = any version from 5.2.1 up to (but not including) 6.0.0

The lock file pins the *exact* version. The `^` just defines the *range* for updates.

## Common Scenarios

### "I just pulled code and things are broken"
```bash
pnpm install   # Someone probably added/updated a dependency
```

### "I need a library for X"
```bash
# Search on https://www.npmjs.com first, then:
pnpm add <package-name>
```

### "My node_modules seems corrupted"
```bash
rm -rf node_modules
pnpm install
```

## Common Gotchas

| Gotcha | Fix |
|--------|-----|
| `ERR_PNPM_OUTDATED_LOCKFILE` | Run `pnpm install` to update the lock file |
| `command not found: pnpm` | Install it: `npm install -g pnpm` |
| Package not found in workspace | Use `--filter` to target the right sub-project |
| Peer dependency warnings | Usually safe to ignore, but read them if something breaks |
